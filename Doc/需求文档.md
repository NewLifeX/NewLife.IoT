# NewLife.IoT 需求文档

## 1. 项目定位

NewLife.IoT 是物联网标准库，为 IoT 领域定义通信协议标准规范，**不含具体协议实现**。目标是成为 IoT 平台与硬件驱动之间的统一契约层，由两类消费者共同使用：

- **IoT 平台**（星尘/ZeroIoT/FIoT 及自定义平台）依赖此库的接口与抽象基类
- **驱动库**（NewLife.Modbus、NewLife.Siemens、NewLife.IoTSerial 等）实现此库的接口

**设计目标**：
- 统一驱动接口，平台端只调用 `IDriver` 一套接口，无需感知底层协议差异
- 异步优先，天然支持并发管理多条链路
- 同时支持轮询型（主动读取）和推送型（设备主动上报）两种数据获取模式
- 结构化结果类型，携带质量码、时间戳、可选诊断帧
- 物模型（ThingSpec）支持，驱动可描述自身的属性、事件、服务规范
- 目标框架覆盖 `net45` ~ `net10.0`，满足工业现场低版本兼容要求

---

## 2. 用户角色

| 角色 | 说明 | 核心诉求 |
|------|------|---------|
| IoT 平台开发者 | 接入多种硬件驱动的平台端开发者 | 统一接口调用，无需感知协议细节；异步并发读写 |
| 驱动开发者 | 实现具体协议的库/插件开发者 | 简单继承基类即可完成驱动，不关心传输层实现 |
| 硬件厂商 | 提供设备 SDK 的第三方厂商 | 通过实现 `IDriver` 接入 IoT 平台，参考物模型 |
| 测试工程师 | 对驱动和平台进行集成验证的工程师 | Mock 驱动接口、模拟设备节点 |

## 3. 功能需求

### 3.1 统一异步驱动接口

- **描述**：`IDriver` 是平台与驱动的唯一契约接口，所有 I/O 方法均为异步
- **用户故事**：作为平台开发者，我希望只调用一套接口（不判断接口类型），以便并发管理多条链路
- **设计要点**：
  - `OpenAsync`/`CloseAsync` 建立/断开与设备的连接，返回/释放 `INode` 节点对象
  - `ReadAsync(INode, IPoint[], CancellationToken)` 批量读取点位，返回 `Task<ReadResult>`（含点位数组、质量码、时间戳和可选诊断帧）
  - `WriteAsync(INode, WriteRequest, CancellationToken)` 单点写入，返回 `Task<WriteResult>`（含回显值和诊断帧）
  - `WriteAsync(INode, WriteRequest[], CancellationToken)` 批量写入，返回 `Task<WriteResult>`（`AffectedCount` 为实际写入数）
  - `ControlAsync(INode, ControlRequest, CancellationToken)` 调用设备服务，返回 `Task<ControlResult>`（含输出参数字典）
  - `CreateParameter`/`GetSpecification` 元数据方法保持同步（无 I/O）
  - `DriverExtensions` 提供同步便利包装
  - 驱动统一继承 `DriverBase`，不再有独立异步驱动接口
- **优先级**：Must

### 3.2 推送事件模型

- **描述**：推送型驱动（MQTT/WebSocket/OPC-UA）在收到设备数据时，通过 `DataReceived` 事件主动通知平台，无需平台轮询
- **用户故事**：作为平台开发者，我希望对推送设备使用统一事件接口，而不是轮询 Read
- **设计要点**：
  - `DataReceived` 事件直接定义在 `IDriver` 接口上，所有驱动均具备
  - 事件参数 `DriverDataEventArgs` 包含 `Node`、`Points`、`Result`（含时间戳和质量码），`Timestamp` 便利属性取自 `Result`
  - `DriverBase.RaiseDataReceived` 提供受保护的触发辅助方法，推送型驱动在收到数据时调用
  - 轮询型驱动无需主动触发该事件（空实现即可）
- **优先级**：Must

### 3.3 结构化结果类型

- **描述**：以类型化结果替代裸字典和裸异常，统一携带成功/失败状态、错误码和诊断数据
- **设计要点**：
  - `IDriverResult` 接口定义 `IsSuccess`/`Code`/`Message`，`ReadResult`/`WriteResult`/`ControlResult` 均实现
  - `ReadResult`：携带 `Points[]`/`Values[]`（平行数组）、`Quality`、`Timestamp`、可选 `RequestBytes`/`ResponseBytes`
  - `WriteResult`（`readonly struct`，零堆分配）：携带 `AffectedCount`、`EchoValue`、可选诊断帧，静态工厂 `Success`/`SuccessBatch`/`Fail`
  - `ControlResult`：携带 `OutputParameters` 字典，静态工厂 `Success`/`Fail`
  - `DataQuality` 枚举：`Good`/`Uncertain`/`Bad`/`BadNotConnected`/`BadTimeout`/`BadChecksumError`/`BadProtocolError`
  - `IDriver.Diagnostics` 属性开关，开启后填充 `RequestBytes`/`ResponseBytes`；生产环境默认关闭，零额外分配
- **优先级**：Must

### 3.4 结构化异常与错误码

- **描述**：定义 IoT 专用异常类型与错误码枚举，区分配置错误与运行时预期失败
- **设计要点**：
  - `IoTException` 继承 `Exception`，包含 `Code: IoTErrorCode`
  - `IoTErrorCode` 枚举（三段分类）：
    - 1000 段（配置/编程错误）：`InvalidParameter`/`DriverNotFound`/`NotSupported`
    - 2000 段（运行时通信错误）：`ConnectionFailed`/`Timeout`/`DeviceBusy`/`ChecksumError`/`ProtocolError`/`ReadError`/`WriteError`
    - 3000 段（设备状态）：`DeviceOffline`/`InvalidAddress`
  - 1000 段错误属于编程/配置问题，抛出 `IoTException`；2000/3000 段在工业现场高频发生，以 `Result.IsSuccess=false` 返回，减少 GC 开销
- **优先级**：Must

### 3.5 写入与服务调用请求对象

- **描述**：用类型化请求对象携带目标点位、写入值及可扩展元数据，便于接口演进不破坏兼容性
- **设计要点**：
  - `WriteRequest`（实现 `IExtend`）：构造函数支持 `()` / `(value)` / `(point, value)`，`Items` 扩展字典存 QoS、优先级等控制元数据
  - `ControlRequest`（实现 `IExtend`）：`ServiceName` + `Parameters` 字典，静态工厂 `ControlRequest.Create(name, parameters)`
- **优先级**：Must

### 3.6 设备发现接口

- **描述**：支持驱动扫描发现网络或总线上的子设备，适用于 Modbus/BACnet 等支持广播扫描的协议
- **设计要点**：
  - `IDiscoverableDriver` 接口定义 `DiscoverAsync(parameters, cancellationToken)`，返回 `IEnumerable<IDeviceInfo>`
  - 平台在组态阶段调用，自动发现总线上的设备列表
  - 不支持扫描的驱动无需实现此接口
- **优先级**：Could

### 3.7 驱动工厂

- **描述**：提供驱动注册、按名称创建实例、按标识符共享单实例的工厂能力
- **设计要点**：
  - `DriverFactory.Register<T>` 注册驱动类型
  - `DriverFactory.Create(name, identifier)` 按名称创建实例，同一 `identifier` 返回共享实例（多设备节点对应同一物理连接）
  - `DriverFactory.CreateParameter(name, parameter)` 创建驱动参数对象
  - `[DriverAttribute]` 标注驱动名称，支持 `DriverFactory` 自动扫描
- **优先级**：Must

## 4. 非功能需求

- **性能**：`WriteResult` 为 `readonly struct`，零堆分配；`ReadResult.Values` 平行数组避免字典哈希开销；`Diagnostics` 默认关闭，生产环境无额外内存分配；热路径不使用反射
- **安全**：驱动参数传递前须校验（非空、范围）；对外异常不暴露内部路径；序列化参数剥离 BOM
- **兼容性**：目标框架 `net45`/`net461`/`net462`/`netstandard2.0`/`netstandard2.1`/`net6+`~`net10.0`；`Task.CompletedTask` 在 net45 用条件编译规避；不使用 `ArgumentNullException.ThrowIfNull` 等高版本 BCL 专属 API

---

## 5. 边界与约束

**不做**：
- 具体协议实现（Modbus、Siemens、OPC-UA 由各自仓库负责）
- IoT 平台上层（ZeroIoT/星尘/FIoT 由产品仓库负责）
- 数据存储、持久化
- 跨语言通信框架（HSL 风格）

**已知限制**：
- 驱动各自管理底层连接（串口、TCP 等），连接管理逻辑由各驱动包内部实现，不在标准库约束范围内

## 7. 术语表

| 术语 | 定义 |
|------|------|
| IDriver | 协议驱动顶级接口，物联网平台与驱动库之间的统一契约 |
| INode | 驱动节点，一条通信链路上的逻辑分片（如 Modbus 站号）；`IsConnected` 反映连接健康状态 |
| IDevice | 逻辑设备，物联网平台的设备对象，接收驱动上报数据 |
| IPoint | 点位，设备上一个可读/写的物理量（名称+地址+类型+长度） |
| ThingSpec | 物模型（TSL），描述设备的属性、事件、服务规范 |
| ReadResult | 读取操作结果，包含 Points/Values 平行数组、DataQuality 质量码、可选诊断帧 |
| WriteResult | 写入操作结果（readonly struct），包含 AffectedCount、回显值、可选诊断帧 |
| ControlResult | 服务调用结果，包含 OutputParameters 输出参数字典 |
| IDriverResult | 驱动操作结果公共接口，统一 IsSuccess/Code/Message 语义 |
| WriteRequest | 写入请求，包装目标点位、值及扩展元数据（IExtend） |
| ControlRequest | 服务调用请求，包含服务名和输入参数字典（IExtend） |
| DataQuality | 数据质量码枚举（参考 OPC UA），Good/Uncertain/Bad 三级，Bad 有多个子码 |
| IoTException | IoT 专用异常，携带 `IoTErrorCode` 便于分类处理 |
| IoTErrorCode | 三段错误码：1000配置错误、2000运行时错误、3000设备状态 |
| DriverBase | 驱动抽象基类，提供 Diagnostics、ServiceProvider、RaiseDataReceived 等公共实现 |
| DriverFactory | 驱动工厂，按名称注册/创建驱动实例，同一标识符保证单实例共享 |
| IDataPort | 历史遗留底层接口，仅做字节读写；新驱动不依赖此接口 |
