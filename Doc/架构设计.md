# NewLife.IoT 架构设计

## 1. 架构概览

### 1.1 整体分层

```
┌───────────────────────────────────────────────────────────────────┐
│                      IoT 平台层  (IDevice)                         │
│   PostProperty / WriteEvent / RegisterService / SetOnline         │
│   ZeroIoT / 星尘 / FIoT / 自定义平台                              │
├───────────────────────────────────────────────────────────────────┤
│                  驱动宿主层  (DriverFactory + INode)               │
│   驱动发现（[Driver] 特性扫描）/ 实例缓存 / 参数序列化             │
├──────────────────┬────────────────────────────────────────────────┤
│   IDriver                                                         │
│   ├── 轮询 I/O   ReadAsync/WriteAsync/ControlAsync                │
│   ├── 推送事件   DataReceived（MQTT/WebSocket/OPC-UA 触发）        │
│   └── 设备发现   IDiscoverableDriver（Modbus/BACnet 扫描）         │

│                     物理设备层                                     │
│   串口设备 / TCP PLC / UDP 传感器 / HTTP API / 数据库              │

```

## 2. 核心接口设计

### 2.1 IDriver — 统一驱动接口

**设计原则**：异步优先，推送事件与轮询使用同一接口，元数据方法保持同步。

```csharp
public interface IDriver
{
    // 推送事件（轮询驱动无需触发）
    event EventHandler<DriverDataEventArgs> DataReceived;

    // 元数据（同步，无 I/O）
    IDriverParameter? CreateParameter(String? parameter = null);
    ThingSpec? GetSpecification();

    // I/O 操作（全部异步）
    Task<INode>         OpenAsync   (IDevice device, IDriverParameter? parameter, CancellationToken cancellationToken = default);
    Task                CloseAsync  (INode node, CancellationToken cancellationToken = default);
    Task<ReadResult>    ReadAsync   (INode node, IPoint[] points, CancellationToken cancellationToken = default);
    Task<WriteResult>   WriteAsync  (INode node, WriteRequest request, CancellationToken cancellationToken = default);
    Task<WriteResult>   WriteAsync  (INode node, WriteRequest[] requests, CancellationToken cancellationToken = default);
    Task<ControlResult> ControlAsync(INode node, ControlRequest request, CancellationToken cancellationToken = default);
}
```

**接口设计要点**：
- `DataReceived` 直接定义在 `IDriver` 上，平台无需 `is ISubscribableDriver` 判断即可统一订阅
- `ReadAsync` 返回 `ReadResult`（含质量码/时间戳/诊断帧），而非裸字典
- `WriteAsync` 批量重载返回 `WriteResult`（含 `AffectedCount`），平台可知道实际写入成功数
- `ControlAsync` 以 `ControlRequest` 封装服务名和参数，返回 `ControlResult`（含 `OutputParameters`）

### 2.2 DriverBase — 驱动抽象基类

```csharp
public abstract class DriverBase : DisposeBase, IDriver, ILogFeature, ITracerFeature
{
    // 服务提供者，可取外部注入对象（如 IBoard 串口映射）
    public IServiceProvider? ServiceProvider { get; set; }

    // 诊断开关：开启后 ReadResult/WriteResult 填充原始帧字节（生产关闭）
    public Boolean Diagnostics { get; set; }

    // 推送辅助：推送型驱动在收到数据时调用
    protected void RaiseDataReceived(INode node, IPoint[] points, ReadResult result);

    // 物模型虚钩子：派生类重写此方法填充物模型内容
    protected virtual Boolean OnGetSpecification(ThingSpec thingSpec) => false;

    // 参数虚钩子：泛型版 DriverBase<TNode,TParameter> 重写此方法
    protected virtual IDriverParameter? OnCreateParameter() => null;
}

// 泛型便利版：固定节点类型和参数类型
public class DriverBase<TNode, TParameter> : DriverBase
    where TNode : INode, new()
    where TParameter : IDriverParameter, new() { }
```

**驱动开发的典型模式**：继承 `DriverBase<TNode, TParameter>`，重写 `OpenAsync`/`ReadAsync` 等虚方法，专注协议解析，连接管理细节由 `TNode` 状态和 `ServiceProvider` 辅助解决。

### 2.3 推送事件模型

```csharp
// 直接定义在 IDriver 上
event EventHandler<DriverDataEventArgs> DataReceived;

public class DriverDataEventArgs : EventArgs
{
    public INode      Node      { get; init; } = null!;
    public IPoint[]   Points    { get; init; } = [];
    public ReadResult Result    { get; init; } = null!;
    public DateTime   Timestamp => Result?.Timestamp ?? DateTime.UtcNow;
}
```

**设计决策**：将 `DataReceived` 合并到 `IDriver`（而非独立 `ISubscribableDriver`）：
- 平台无需判断 `if (driver is ISubscribableDriver)` 即可统一订阅
- 轮询驱动的 `DataReceived` 永不触发，无额外开销
- `DriverDataEventArgs` 携带完整 `ReadResult`（含质量码），比旧版裸字典更丰富

### 2.4 结构化结果类型

#### IDriverResult（公共接口）

```csharp
public interface IDriverResult
{
    Boolean     IsSuccess { get; }
    IoTErrorCode Code     { get; }
    String?     Message   { get; }
}
```

#### ReadResult

```csharp
public class ReadResult : IExtend, IDriverResult
{
    public Boolean      IsSuccess  { get; init; }
    public IoTErrorCode Code       { get; init; }
    public String?      Message    { get; init; }
    public DateTime     Timestamp  { get; init; } = DateTime.UtcNow;
    public DataQuality  Quality    { get; init; } = DataQuality.Good;
    public IPoint[]     Points     { get; init; } = [];    // 与 Values 平行
    public Object?[]    Values     { get; init; } = [];
    public Byte[]?      RequestBytes  { get; init; }       // Diagnostics=true 时填充
    public Byte[]?      ResponseBytes { get; init; }
}
```

#### WriteResult（零堆分配）

```csharp
public readonly struct WriteResult : IDriverResult
{
    public Boolean      IsSuccess     { get; init; }
    public IoTErrorCode Code          { get; init; }
    public String?      Message       { get; init; }
    public Int32        AffectedCount { get; init; }   // 单点=1，批量=成功数
    public Object?      EchoValue     { get; init; }   // 协议回显值
    public Byte[]?      RequestBytes  { get; init; }
    public Byte[]?      ResponseBytes { get; init; }

    public static WriteResult Success(...)    { }
    public static WriteResult SuccessBatch(Int32 count) { }
    public static WriteResult Fail(IoTErrorCode code, String message) { }
}
```

#### ControlResult

```csharp
public class ControlResult : IExtend, IDriverResult
{
    public Boolean      IsSuccess        { get; init; }
    public IoTErrorCode Code             { get; init; }
    public String?      Message          { get; init; }
    public IDictionary<String, Object?> OutputParameters { get; init; }

    public static ControlResult Success(IDictionary<String, Object?>? output = null) { }
    public static ControlResult Fail(IoTErrorCode code, String message) { }
}
```

### 2.5 DataQuality — 数据质量码

```csharp
public enum DataQuality : Byte
{
    Good             = 0,   // 数据有效可信
    Uncertain        = 1,   // 数据可能有效（如断线恢复后的最后已知值）
    Bad              = 2,   // 采集失败
    BadNotConnected  = 3,   // 设备离线或连接未建立
    BadTimeout       = 4,   // 通信超时
    BadChecksumError = 5,   // 响应校验失败
    BadProtocolError = 6,   // 响应格式不符合协议
}
```

参考 OPC UA 质量码体系，驱动在失败时应填充具体子码（Bad 子码）而非仅返回 Bad。

### 2.7 IoTException + IoTErrorCode

```csharp
public enum IoTErrorCode
{
    None = 0,
    // 1000段：配置/编程错误 → 应抛出，不被 TryRead 捕获
    InvalidParameter = 1001,
    DriverNotFound   = 1002,
    NotSupported     = 1003,
    // 2000段：运行时通信错误 → 频繁发生，应用结果类型
    ConnectionFailed = 2001,
    Timeout          = 2002,
    DeviceBusy       = 2003,
    ChecksumError    = 2004,
    ProtocolError    = 2005,
    ReadError        = 2006,
    WriteError       = 2007,
    // 3000段：设备状态
    DeviceOffline    = 3001,
    InvalidAddress   = 3002,
}
```

## 3. 数据模型

### 3.1 物模型 ThingSpec

```
ThingSpec
├── Profile（产品元数据：ProductKey/Version）
├── Properties[]（属性规范：Identifier/Name/AccessMode/DataType/Address/Scaling）
├── Events[]（事件规范：Identifier/Type=info|alert|error/OutputData）
└── Services[]（服务规范：Identifier/Type=sync|async/InputData/OutputData）
```

### 3.2 IPoint（点位）

```
IPoint
├── Name（点位名称，对应 ThingSpec.Properties[].Identifier）
├── Address（驱动特定地址，如 4x0001/DI07/D100）
├── Type（数据类型字符串：int32/float/bool/string 等）
└── Length（字节数或字符串长度）
```

### 3.3 INode（节点）

```csharp
public interface INode
{
    IDriver           Driver      { get; set; }
    IDevice?          Device      { get; set; }
    IDriverParameter? Parameter   { get; set; }
    Boolean           IsConnected { get; }   // 长连接驱动在断线时置 false
}
```

`IsConnected` 允许平台在不触发一次采集的情况下感知连接健康状态。轮询驱动默认返回 `true`；TCP/串口驱动在断线时置为 `false`。

### 3.4 节点关系

```
IDriver（1）──────────────────（N）INode
                                    │
IDevice（1）─────────────────────── │
                                    │
IDriverParameter（1）──────────── ──│
```

同一物理链路（如 COM1 串口）可挂载多个逻辑设备（INode），各节点共用驱动内部的底层连接（驱动自行管理）。

## 4. 技术选型

| 领域 | 选型 | 理由 |
|------|------|------|
| 异步模型 | `Task<T>` + `CancellationToken` | .NET 标准模式，net45+ 全覆盖 |
| 读取结果 | `ReadResult`（class，含质量码） | 质量码/诊断帧/时间戳内置，避免散落字段 |
| 写入结果 | `WriteResult`（readonly struct） | 零堆分配，单点/批量通用 |
| 推送通知 | `EventHandler<DriverDataEventArgs>` | 合并到 IDriver，无需 is 判断，NewLife 风格 |
| 序列化 | XML 优先（注释齐全）/ JSON 备选 | 参数可读性，与 NewLife 现有约定一致 |
| 目标框架 | `net45~net10.0` + `netstandard2.0/2.1` | 工业设备低版本兼容要求 |
| 传输层 | 各驱动包自行实现，不在标准库约束 | 串口/TCP/UDP/HTTP 差异大，无法统一封装 |

## 5. 关键设计决策

| 决策点 | 选定方案 | 备选方案 | 选择理由 |
|--------|---------|---------|---------|
| `DataReceived` 位置 | 直接在 `IDriver` 上 | 独立 `ISubscribableDriver` | 平台无需 is 判断，接口统一 |
| `ReadAsync` 返回类型 | `ReadResult`（含质量码） | 裸 `IDictionary<String, Object?>` | 质量码/时间戳/诊断帧内置，避免平台自行包装 |
| 批量写入返回 | `Task<WriteResult>`（含 AffectedCount） | `Task`（无返回） | 平台可知道实际成功数 |
| 请求对象化 | `WriteRequest`/`ControlRequest` | 直接传值/裸字典 | IExtend 扩展元数据，接口演进不破坏兼容性 |
| 诊断帧 | `Diagnostics` 开关 + 可选字段 | 固定填充 | 生产环境零额外分配，按需开启 |
| `INode.IsConnected` | 驱动维护，轮询默认 true | 触发采集才能感知 | 平台可主动感知连接健康，无需空读 |
| 传输层封装 | 不在标准库定义 | 统一 ITransport 接口 | 各协议差异大，封装层无实际价值 |

## 6. 已知约束

- 驱动各自管理底层连接，连接管理策略（重连间隔、超时等）不在标准库契约范围内
- `IDataPort` 作为历史遗留接口保留，不影响 `IDriver` 体系
- net45 缺少 `Task.CompletedTask`，需条件编译规避；不使用 `ArgumentNullException.ThrowIfNull` 等高版本专属 API
- `WriteResult` 为 `readonly struct`，通过 `IDriverResult` 接口存储时会装箱，热路径应使用具体类型追踪
