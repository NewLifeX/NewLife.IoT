# NewLife.IoT 架构设计

> 版本：v3.1 | 日期：2026-07-22

本文档描述 NewLife.IoT 的架构分层、核心组件和数据流。需求定义见[需求文档](需求文档.md)，实现状态见[功能清单](功能清单.md)。

## 1. 架构概览

### 1.1 分层架构

```
┌──────────────────────────────────────────────────────────────────────┐
│                     IoT 平台层  (IDevice)                             │
│  PostProperty / PostEvent / RegisterService / SetOnline / SetOffline │
│  ZeroIoT / 星尘 / FIoT / 自定义平台                                  │
├──────────────────────────────────────────────────────────────────────┤
│                  驱动宿主层  (DriverFactory + INode)                  │
│  插件扫描（[Driver] 特性）/ 实例缓存（identifier 单例）/ 参数序列化   │
├──────────────────┬───────────────────────────────────────────────────┤
│  IDriver                                                             │
│  ├── 轮询 I/O   ReadAsync/WriteAsync/ControlAsync                   │
│  ├── 推送事件   DataReceived（MQTT/WebSocket/OPC-UA 触发）           │
│  ├── 传输抽象   ITransport（Open/Close/Send/Receive/SendReceive）    │
│  └── 设备发现   IDiscoverableDriver（Modbus/BACnet 扫描）            │
├──────────────────────────────────────────────────────────────────────┤
│                      物理设备层                                       │
│  串口设备 / TCP PLC / UDP 传感器 / HTTP API / 数据库                  │
└──────────────────────────────────────────────────────────────────────┘
```

**各层职责**：

| 层 | 包/项目 | 职责 | 不负责 |
|----|---------|------|--------|
| 平台层 | 产品仓库（ZeroIoT 等） | 设备生命周期管理、数据持久化、规则引擎、可视化 | — |
| 宿主层 | `NewLife.IoT` | 驱动注册发现、实例缓存、参数序列化 | 具体协议实现 |
| 驱动层 | `NewLife.IoT`（接口）+ 各驱动包（实现） | 统一 I/O 契约、结果类型、传输抽象 | 传输层具体实现 |
| 传输层 | `IoTSerial`/`IoTSocket`/`IoTDatabase` | 串口/TCP/UDP/HTTP/DB 的连接管理与数据收发 | 协议编解码 |
| 设备层 | 物理硬件 | 响应协议指令 | — |

## 2. 核心组件

### 2.1 IDriver — 统一驱动接口

**设计原则**：异步优先，推送事件与轮询使用同一接口，元数据方法保持同步。

核心方法签名：

| 方法 | 返回 | 说明 |
|------|------|------|
| `OpenAsync(device, parameter, ct)` | `Task<INode>` | 建立连接，返回节点 |
| `CloseAsync(node, ct)` | `Task` | 断开连接，多节点共用时最后关闭 |
| `ReadAsync(node, points[], ct)` | `Task<ReadResult>` | 批量读取点位 |
| `WriteAsync(node, requests[], ct)` | `Task<WriteResult>` | 批量写入，`AffectedCount` 为实际成功数 |
| `ControlAsync(node, request, ct)` | `Task<ControlResult>` | 调用设备服务 |
| `DataReceived` | `event` | 推送型驱动数据到达事件 |
| `CreateParameter(parameter?)` | `IDriverParameter?` | 创建/解析驱动参数（同步） |
| `GetSpecification()` | `ThingSpec?` | 获取物模型（同步） |

**关键决策**：`WriteAsync` 只保留批量重载。单点写入通过 `DriverExtensions.WriteAsync(node, singleRequest)` 扩展包装为单元素数组后调用批量方法。驱动实现者只需关注批量路径。

### 2.2 DriverBase — 驱动抽象基类

```csharp
public abstract class DriverBase : DisposeBase, IDriver, ILogFeature, ITracerFeature
{
    public IServiceProvider? ServiceProvider { get; set; }  // DI 容器
    public Boolean Diagnostics { get; set; }                // 诊断开关

    protected void RaiseDataReceived(INode, IPoint[], ReadResult);  // 推送触发
    protected virtual Boolean OnGetSpecification(ThingSpec) => false;
    protected virtual IDriverParameter? OnCreateParameter() => null;

    // 虚方法（派生类重写实现具体协议）
    public virtual Task<INode> OpenAsync(...) => TaskEx.FromResult(new Node {...});
    public virtual Task<ReadResult> ReadAsync(...) => throw new NotImplementedException();
    public virtual Task<WriteResult> WriteAsync(...) => throw new NotImplementedException();
    public virtual Task<ControlResult> ControlAsync(...) => throw new NotImplementedException();
}

// 泛型版 — 固定节点类型和参数类型
public class DriverBase<TNode, TParameter> : DriverBase
    where TNode : INode, new()
    where TParameter : IDriverParameter, new()
```

**驱动开发典型模式**：继承 `DriverBase<TNode,TParameter>` → 重写 `OpenAsync`/`ReadAsync`/`WriteAsync`/`ControlAsync` → 专注协议编解码，连接管理由 `TNode` 状态和 `ITransport`/`ServiceProvider` 辅助。

### 2.3 推送事件模型

```
设备数据到达 → 驱动调用 RaiseDataReceived(node, points, result)
    → 触发 IDriver.DataReceived 事件
    → 平台在事件处理中收到 DriverDataEventArgs { Node, Points, Result }
    → Result 含 Timestamp + Quality，平台无需额外时间戳处理
```

`DataReceived` 直接定义在 `IDriver` 上（非独立 `ISubscribableDriver`），平台统一订阅，轮询驱动永不触发。

### 2.4 结构化结果

```
IDriverResult
├── IsSuccess: Boolean
├── Code: IoTErrorCode
└── Message: String?

ReadResult : IExtend, IDriverResult        WriteResult(struct) : IDriverResult
├── Timestamp                              ├── AffectedCount
├── Quality: DataQuality                   ├── EchoValue
├── Points[] / Values[] (平行数组)          ├── RequestBytes
├── RequestBytes / ResponseBytes           └── ResponseBytes
└── GetValue(name) 按名称查找              + Success() / SuccessBatch(n) / Fail()

ControlResult : IExtend, IDriverResult
├── OutputParameters
└── + Success(output?) / Fail()
```

**核心设计**：`Points[]` 与 `Values[]` 为平行数组（同下标对应），避免字典哈希开销。`WriteResult` 为 `readonly struct` 零堆分配。`GetValue(name)` 按名称或地址查找值。

### 2.5 错误处理策略

```
IoTErrorCode
├── 1000 段（配置/编程错误）→ 抛出 IoTException
│   InvalidParameter / DriverNotFound / NotSupported
├── 2000 段（运行时通信错误）→ Result.IsSuccess=false
│   ConnectionFailed / Timeout / ChecksumError / ProtocolError / ReadError / WriteError
└── 3000 段（设备状态）→ Result.IsSuccess=false
    DeviceOffline / InvalidAddress
```

高频采集场景中，2000/3000 段错误以结果类型返回而非抛异常，减少 GC 压力。

### 2.6 ITransport — 传输层抽象

```csharp
public interface ITransport : IDisposable
{
    Boolean IsConnected { get; }
    Int32 Timeout { get; set; }

    void Open();
    void Close();
    void Send(ReadOnlySpan<Byte> data);
    Byte[]? Receive(Int32 timeout = -1);
    Byte[]? SendReceive(ReadOnlySpan<Byte> request, Int32 timeout = -1);  // 原子请求-响应
}
```

**实现映射**：

| 接口 | 串口 | TCP | UDP | HTTP |
|------|------|-----|-----|------|
| `ITransport` | `SerialTransport` | `TcpTransport` | `UdpTransport` | `HttpTransport` |
| 所在包 | `NewLife.IoTSerial` | `NewLife.IoTSocket` | `NewLife.IoTSocket` | `NewLife.IoTSocket` |

`SendReceive` 保证请求-响应原子性，多线程自动串行化。

### 2.7 ThingSpec — 物模型

```
ThingSpec
├── Schema: "http://iot.feifan.link/schema.json"
├── Profile { Version, ProductKey }
├── Properties[] → PropertySpec { Id, Name, AccessMode, DataType { Type, Specs { min/max/step/unit/mapping } }, Address, Scaling, ReadRule, WriteRule }
├── Events[]    → EventSpec { Id, Name, Type(info|warning|error), Method, OutputData[] }
└── Services[]  → ServiceSpec { Id, Name, Type(sync|async), Method, InputData[], OutputData[] }
```

序列化路径：`FromJson(json)` 用 NewLife `JsonParser` → `ToJson()` 用 `JsonWriter`（CamelCase + EnumString），保证往返一致性。

### 2.8 INode — 节点关系

```
IDriver (1) ──── (N) INode
                      │
IDevice (1) ──────────┤
                      │
IDriverParameter (1) ─┘
```

同一物理链路（COM1 串口 / 192.168.1.1:502）可挂载多个 `INode`。`INode.IsConnected` 由驱动维护，平台可主动感知连接健康。

### 2.9 DriverFactory — 插件体系

```
[Driver("ModbusTcp")]           ← 驱动标注名称
    ↓
DriverFactory.Register<T>()     ← 手动注册
DriverFactory.ScanAll()         ← 自动扫描所有程序集
    ↓
_map["ModbusTcp"] = typeof(ModbusTcpDriver)
    ↓
DriverFactory.Create("ModbusTcp", "192.168.1.1:502")
    → 同一 identifier 返回缓存单例（多设备共享物理连接）
    → 无 identifier 时每次新建实例
    ↓
DriverFactory.CreateParameter("ModbusTcp", xml)
    → 创建驱动参数对象，解析 Xml/Json 配置
```

### 2.10 控制器接口（CTL 模块）

物理硬件控制抽象层，服务于嵌入式/工业场景：

| 接口 | 用途 | 典型实现 |
|------|------|---------|
| `IBoard` | 板卡设备名映射 + 工厂 | A2 工业计算机 COM1→/dev/ttyAMA0 |
| `ISerialPort` | 串口打开/关闭/读写 | `DefaultSerialPort`（System.IO.Ports） |
| `IInputPort` / `IOutputPort` | 数字 I/O 读写 | `FileInputPort` / `FileOutputPort` |
| `IModbus` | Modbus 功能码原语 | 驱动层内部使用，直接操作线圈/寄存器 |
| `IRelayController` | 继电器控制 | 基于 `IModbus` 操作 |

这些接口通过 `DriverBase.ServiceProvider` 注入到驱动中，驱动无需关心具体硬件平台差异。

## 3. 技术选型

| 领域 | 选型 | 理由 |
|------|------|------|
| 异步模型 | `Task<T>` + `CancellationToken` | .NET 标准，net45+ 全覆盖 |
| 读取结果 | `ReadResult`（class，含 Quality） | 质量码/时间戳/诊断帧内置 |
| 写入结果 | `WriteResult`（readonly struct） | 零堆分配，单点/批量通用 |
| 推送通知 | `EventHandler<DriverDataEventArgs>` | 合并到 IDriver，无需 `is` 判断 |
| 序列化 | XML 优先（注释齐全）/ JSON 备选 | 参数可读，与 NewLife 约定一致 |
| 目标框架 | `net45`~`net10.0` + `netstandard2.0/2.1` | 工业低版本兼容 |
| 传输层 | `ITransport` 在标准库定义 | 统一契约，实现在子项目中 |

## 4. 关键设计决策

| 决策点 | 选定方案 | 备选方案 | 理由 |
|--------|---------|---------|------|
| `DataReceived` 位置 | 直接在 `IDriver` | 独立 `ISubscribableDriver` | 平台无需 `is` 判断 |
| `ReadAsync` 返回 | `ReadResult`（含质量码） | 裸字典 | 质量码/时间戳/诊断帧内置 |
| 批量写入 | `WriteAsync(requests[])` | 单点重载 | 支持批量帧的驱动可一次性发送 |
| 请求对象化 | `WriteRequest`/`ControlRequest` + `IExtend` | 裸值/字典 | 接口演进不破坏兼容性 |
| 诊断帧 | `Diagnostics` 开关 + 可选字段 | 固定填充 | 生产环境零额外分配 |
| `INode.IsConnected` | 驱动维护，轮询默认 true | 触发采集感知 | 平台主动感知连接健康 |
| 传输层 | `ITransport` 在标准库 | 各驱动自行管理 | 统一契约，子项目实现 |
| CTL 模块归属 | 独立模块（硬件控制抽象） | 合并到 UTL | 物理硬件与纯工具类本质不同 |

## 5. 已知约束

- **IDataPort**：v1.x 历史遗留接口，v3.0 起源码已移除，新驱动不依赖。
- **net45 兼容**：`Task.CompletedTask` 用 `TaskEx.CompletedTask` 条件编译；禁用 `ArgumentNullException.ThrowIfNull`。
- **WriteResult 装箱**：通过 `IDriverResult` 接口存储时会装箱，热路径应使用具体类型。
- **IDictionarySourceConverter**：`PropertySpec` 通过 System.Text.Json 反序列化受限，物模型解析推荐使用 `ThingSpec.FromJson()`（NewLife 原生解析器）。
